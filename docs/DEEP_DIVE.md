# Tailscale + Shadowrocket + macOS 原生 SSH 自愈方案

## 1. 这份资料解决什么问题

这份资料记录的是一个在 macOS 上很容易出现、但不容易第一次就定位清楚的问题：

- Tailscale 已经连上
- Shadowrocket 也在运行
- 远端 Linux 服务器在 tailnet 里显示在线
- 但通过原生 SSH 访问这台 Mac 时，连接会突然失效
- 重新执行一次 `tailscale down` 和 `tailscale up` 后，SSH 往往又恢复

这份资料的目标不是只给出一个“能用的脚本”，而是把下面几件事都讲清楚：

- 问题现象是什么
- 背后的网络原理是什么
- 实际排查时应该看哪些信息
- 最终为什么判断问题出在路由归属，而不是 SSH 服务本身
- 为什么最终采用“检测 + 自愈 + 动态退避”的方案

只要读者具备基础的网络知识，例如知道：

- IP 地址
- 路由
- 网卡
- TCP 端口
- SSH 是基于 TCP 的远程登录协议

那么读完这份文档，应该能独立理解并复现整套方案。

---

## 2. 必要背景知识

### 2.1 什么是 Tailscale

Tailscale 可以理解为一个基于 WireGuard 的组网工具。  
它会在多台设备之间建立一个私有覆盖网络，让这些设备即使位于不同局域网、不同 NAT、不同公网环境下，也能像在一个私有网络里一样互相通信。

在这个覆盖网络里，每个节点会拥有一个 Tailscale 地址。  
常见的 IPv4 地址段是：

- `100.64.0.0/10`

例如：

- 本机：`100.96.202.95`
- 远端 Linux：`100.82.42.75`

这些地址不是你的家庭局域网地址，也不是公网地址，而是 Tailscale 在 tailnet 内部分配给各节点的地址。

---

### 2.2 什么是 Tailscale SSH

Tailscale 里有两个容易混淆但必须区分的层次：

第一层是 **Tailscale 网络本身**：

- 负责让两台机器可以互相到达
- 解决的是“路通不通”

第二层是 **Tailscale SSH**：

- 这是 Tailscale 自己提供的一套 SSH 接入机制
- 它会把 SSH 的认证和访问控制接入到 Tailscale 的身份体系里
- 是否启用它，不等于是否使用 Tailscale 网络

也就是说，下面两种场景要分开看：

- 通过 Tailscale 网络访问 **macOS 自带 `sshd`**
- 通过 Tailscale 网络访问 **Tailscale SSH**

本资料解决的是第一种，也就是：

- 用 Tailscale 负责组网
- 用 macOS 原生 SSH 负责登录

---

### 2.3 什么是 macOS 原生 SSH

macOS 自带 OpenSSH 服务端，也就是常说的：

- `sshd`

开启后，它会在系统的 `22` 端口监听 SSH 登录请求。  
在 macOS 上，这个服务通常由 `launchd` 按需拉起，所以你不一定总能看到一个常驻的 `sshd` 进程，但：

- `22` 端口可以处于监听状态
- 真有连接进来时，`launchd` 才临时拉起 `sshd`

---

### 2.4 什么是 Shadowrocket / Packet Tunnel / `utun`

Shadowrocket 在 macOS 上工作时，通常会启用系统的 Network Extension / Packet Tunnel 机制。  
它会创建虚拟隧道接口，例如：

- `utun5`

Tailscale 也会创建类似的虚拟接口，例如：

- `utun0`

这里要特别注意：

- `utun` 的编号不是固定协议的一部分
- `utun0`、`utun5` 只是某次启动时系统分配的接口名
- 在另一台 Mac 上，或者同一台机器的另一次启动中，编号可能变化

因此，**不能把 “Tailscale 一定是 `utun0`” 当成通用规律**。  
更严谨的做法是：

- 先根据本机当前的 Tailscale 地址，找出 Tailscale 实际在用的接口
- 再比较目标流量是不是走到了这个接口

本资料中的脚本已经按这个方式实现。

---

### 2.5 为什么 Tailscale 和 Shadowrocket 会“冲突”

这里的“冲突”不是说：

- WireGuard 和代理协议在协议层面互相不兼容

真正的冲突点是：

- 它们都可能在同一台 Mac 上创建自己的隧道接口
- 它们都可能修改系统路由
- 它们都可能影响 DNS、默认路由、特定网段的归属

所以更准确的表述应该是：

**它们不是协议冲突，而是同一台主机上的流量接管权冲突。**

谁来接管哪些网段、哪些目标地址、哪些 TCP 连接，才是问题的核心。

---

## 3. 这次问题的实际现象

实际现象包括：

- 先启动 Tailscale，再切换 Shadowrocket 节点或开关，之后 SSH 可能失效
- 有时关闭 Shadowrocket 代理后，SSH 也会失效
- 此时 `tailscale status` 往往仍然显示节点在线
- 此时 `tailscale ping` 往往仍然能通
- 但普通的 `ssh user@100.x.x.x` 却连不上
- 重新执行一次 `tailscale down` 再 `tailscale up`，SSH 往往恢复

这些现象放在一起，说明问题很可能不是：

- 远端 Linux 掉线
- 本机 `sshd` 完全没开
- Tailscale 完全断网

而是：

- Tailscale 自己还活着
- 但普通应用的系统路由已经偏离了 Tailscale 的隧道接口

---

## 4. 最终定位到的根因

根因是 **系统路由被 Shadowrocket 的隧道刷新后，Tailscale 地址段不再走 Tailscale 自己的接口**。

具体来说：

- Tailscale 需要把 `100.64.0.0/10` 这类 tailnet 地址送进自己的隧道接口
- 如果 Shadowrocket 在切节点、开关代理或刷新 TUN 状态时重写了路由
- 那么这段流量可能被改发到物理网卡，例如 `en0`

一旦出现这种情况：

- `tailscaled` 自己仍然可能知道怎么联系对端
- 但普通应用（例如 `ssh`）使用的是系统 TCP 栈
- 系统 TCP 栈一旦按错误路由发包，就会导致 SSH 连接失败

所以最准确的根因表述是：

**问题不是 Tailscale 自己完全失效，而是“普通应用使用的系统路由”与“Tailscale 期望的隧道路由”发生了偏离。**

---

## 5. 为什么 `tailscale ping` 能通，但 `ssh` 不能通

这是整个问题里最容易误判的点。

### 5.1 `tailscale ping` 检查的是什么

`tailscale ping` 更接近于在检查：

- `tailscaled` 是否还能找到对端节点
- Tailscale 的点对点链路是否仍然存在
- 当前是否还能通过直接连接或 DERP 中继到达对端

因此，即便系统普通路由已经被改坏，`tailscale ping` 仍然可能成功。

---

### 5.2 原生 SSH 检查的是什么

原生 SSH 使用的是系统 TCP 栈。  
它依赖的是：

- 正确的目标路由
- 正确的虚拟接口归属
- 目标 `22` 端口确实可达

所以只要系统把去 `100.82.42.75` 的流量发到了错误接口，原生 SSH 就会失败。

这也是为什么会出现这种看起来矛盾的情况：

- Tailscale 状态看起来没问题
- 但 SSH 已经坏了

---

## 6. 实际排查时用到的判断方法

### 6.1 看 Tailscale 的节点状态

```bash
tailscale status
tailscale status --json
```

这一步用来确认：

- 节点是否仍然在 tailnet 中
- 对端是否仍然在线
- 当前是否仍然有 direct 或 DERP 路径

这一步很重要，但 **不能单独作为 SSH 是否可用的判断依据**。

---

### 6.2 看系统路由

```bash
route -n get 100.82.42.75
netstat -rn -f inet
```

需要重点看的是：

- 去目标节点的流量到底走了哪个接口
- `100.64.0.0/10` 是否被错误指到了 `en0`

更严谨的判断不是“是否一定走 `utun0`”，而是：

- 是否走到了 **当前 Tailscale 实际使用的那个 `utun` 接口**

因为 `utun` 编号是动态的，不应该在文档或脚本里写死。

---

### 6.3 直接测试 SSH 端口

```bash
nc -G 3 -vz 100.82.42.75 22
```

这一步比 `ping` 更贴近真实需求，因为它直接检测：

- 当前系统 TCP 栈能不能连到对端 SSH 端口

也就是说，它回答的是：

- “SSH 真的能不能用”

而不是：

- “网络层是否还有某种联系”

---

## 7. Shadowrocket 配置做了哪些优化

为了尽量减少 Shadowrocket 接管 Tailscale 流量的概率，配置文件里做了以下处理：

### 7.1 在 `skip-proxy` 中加入：

- `100.64.0.0/10`
- `100.100.100.100`
- `*.ts.net`

### 7.2 在规则顶部加入显式直连规则：

- `DOMAIN-SUFFIX,ts.net,DIRECT`
- `IP-CIDR,100.64.0.0/10,DIRECT,no-resolve`
- `IP-CIDR,100.100.100.100/32,DIRECT,no-resolve`
- `IP6-CIDR,fd7a:115c:a1e0::/48,DIRECT,no-resolve`

### 7.3 删除远程 `update-url`

这样做是为了避免：

- 本地专门针对 Tailscale 的规则被远程订阅覆盖

---

### 7.4 为什么这些优化仍然不能彻底解决问题

因为这类优化主要作用在：

- 规则匹配层
- 代理转发层

但本问题的核心在更底层：

- Packet Tunnel 接口
- 系统路由
- 流量接管顺序

也就是说，即便规则写得对，Shadowrocket 在某些时机仍然可能刷新系统路由，这就是为什么最终还需要自愈机制。

---

## 8. 最终采用的方案为什么是“检测 + 自愈”

最后采用的不是“绝不让 Shadowrocket 改路由”的方案，而是：

1. 定期检查 SSH 路径是否仍然正确
2. 一旦发现异常，就自动重启 Tailscale
3. 如果异常连续发生，就降低检查频率，避免高频重启

这样做的理由是：

- 现实中可操作
- 不依赖手动盯着网络状态
- 能在大多数实际故障发生时快速恢复
- 遇到持续性波动时不会盲目高频重启

---

## 9. 这套脚本分别做什么

目录中有 3 个脚本。

### 9.1 `restart-tailscale.sh`

作用：

- 执行 `tailscale down`
- 执行 `tailscale up`
- 等待几秒后输出当前状态

这是最基础的恢复动作。

---

### 9.2 `check-and-heal-tailscale-ssh.sh`

作用：

- 获取本机当前的 Tailscale IPv4 地址
- 根据这个地址找出 Tailscale 实际在用的接口
- 检查目标主机的路由是否走到了这个接口
- 检查目标主机 `22` 端口是否可达
- 检查 `tailscale ping` 是否仍然能从 Tailscale 控制面角度到达对端
- 根据这些信号对异常做分类
- 只有判断为“本机 Tailscale 或本机路由异常”时才执行重启

这里的退出码设计很关键：

- `0` 表示检查正常
- `10` 表示检查异常，且已经执行了本机自愈动作
- `20` 表示 Tailscale 链路正常，但更像是远端 SSH 服务本身有问题
- `21` 表示路由仍然正常，但 Tailscale 对端暂时不可达，更像是远端离线或更深层网络波动
- 其他退出码表示脚本本身发生错误

---

### 9.3 `watch-tailscale-ssh.sh`

作用：

- 通过 `launchd` 的 `StartInterval` 做成后台定时服务
- 记录状态文件
- 记录中文日志
- 实现动态退避
- 每天跨过 0 点后自动恢复到基础检查间隔
- 对退避后的最大检查间隔做上限控制
- 避免同一时间出现多个并发检查实例
- 自动清理异常退出后可能残留的旧锁
- 让系统在睡眠期间错过检查点后，唤醒时由 `launchd` 补跑一次

它是整套方案的调度器。

---

## 10. 检测原理到底是什么

这套方案检查的不是“Tailscale 活着没有”，而是分三层检查：

### 10.1 路由是否仍然走到 Tailscale 当前接口

也就是：

- 目标节点的流量是否仍然进入了 Tailscale 所在的那个 `utun` 接口

### 10.2 SSH 端口是否仍然真正可达

也就是：

- 从当前系统 TCP 栈出发，是否真的能连到 `100.82.42.75:22`

### 10.3 Tailscale 自己是否仍然能看到对端

也就是：

- `tailscale ping` 是否仍然能够到达对端节点

这样做的原因是：

- 只看 `22` 端口，容易把“远端 SSH 自己挂了”误判为“本机 Tailscale 坏了”
- 只看 `tailscale ping`，又容易忽略“系统普通 TCP 路由已经偏离”这类问题

因此现在的判断逻辑是：

- 路由正常 + `22` 端口正常：认为健康
- 路由异常：认为更像本机路由被抢走，触发重启
- 路由正常 + `22` 端口异常 + `tailscale ping` 正常：认为更像远端 SSH 服务问题，不重启本机
- 路由正常 + `22` 端口异常 + `tailscale ping` 也异常：认为更像远端离线或更深层网络问题，保守处理，不贸然重启本机

只有真正有足够本机证据时，脚本才会自动重启 Tailscale。

这比只看：

- `tailscale status`
- `tailscale ping`

要更接近真实使用场景。

---

## 11. 动态退避策略是怎么工作的

当前逻辑如下：

- 基础检查间隔：`180` 秒，也就是 3 分钟
- 只有“检查异常，并且已经触发 Tailscale 重启”时：
  - 连续异常次数 `+1`
- 当连续异常次数达到偶数，也就是第 2、4、6... 次时：
  - 当前检查间隔翻倍
- 当前实现中，最大检查间隔上限是 `3600` 秒，也就是 1 小时
- 如果某次检查正常，或者虽然异常但没有触发重启：
  - 连续异常次数清零
  - 当前间隔恢复到 180 秒
- 如果已经跨过新的一天：
  - 连续异常次数清零
  - 当前间隔恢复到 180 秒

例如：

- 初始：3 分钟
- 连续第 2 次异常后：6 分钟
- 连续第 4 次异常后：12 分钟
- 连续第 6 次异常后：24 分钟
- 继续翻倍后，最多增长到 60 分钟

只要任意一次检查恢复正常：

- 连续异常次数归零
- 检查间隔立即恢复为 3 分钟

这样做的目的，是在两种情况之间取得平衡：

- 偶发故障时，尽快自愈
- 持续波动时，不靠高频重启硬顶

## 12. 日志记录什么

日志文件路径：

```bash
~/Library/Logs/tailscale-ssh-heal.log
```

日志采用中文分块格式，主要记录：

- 后台巡检启动
- 检查异常并已触发重启
- 检查异常但未触发重启
- 异常时的本机 Tailscale 地址、Tailscale 接口、目标路由接口
- TCP 端口检查结果
- `tailscale ping` 检查结果
- 重启动作已经执行完成
- 连续异常达到两次后触发退避
- 某次检查恢复正常，间隔恢复到基础值
- 脚本自身执行错误

这样做的目的，是让使用者在回看日志时能迅速回答以下问题：

- 什么时候出问题
- 当时是路由错了，还是 TCP 端口不可达
- 是否已经自动重启
- 当前检查频率是多少

---

## 13. 状态文件记录什么

状态文件路径：

```bash
~/Library/Application Support/tailscale-ssh-heal/state.env
```

这个文件记录：

- `CURRENT_INTERVAL`
- `CONSECUTIVE_FAILURES`
- `LAST_RESULT`
- `NEXT_CHECK_AT`
- `LAST_CHECK_AT`
- `LAST_RESET_DAY`

它的作用是：

- 让后台状态可以被查看
- 让退避逻辑可追踪
- 在守护脚本重启后仍能保留最近状态
- 让“当前是否已经到下次检查时间”可被明确判断
- 让“是否已经跨过新的一天并需要恢复基础间隔”可被明确判断

这里还有一个实现上的安全点：

- 脚本现在不会直接 `source state.env`
- 而是只按受控的键值对去解析已知字段

这样即使状态文件被意外写坏，也不会把它当成完整 shell 脚本执行

## 14. 如何安装和使用

以下命令假设你当前位于仓库根目录：

```bash
cd tailscale-shadowrocket-ssh-heal
```

### 14.1 先给脚本执行权限

```bash
chmod +x scripts/restart-tailscale.sh
chmod +x scripts/check-and-heal-tailscale-ssh.sh
chmod +x scripts/watch-tailscale-ssh.sh
```

### 14.2 启动后台巡检

```bash
zsh scripts/watch-tailscale-ssh.sh start
```

默认参数是：

- 目标：`100.82.42.75`
- 端口：`22`
- 基础检查间隔：`180` 秒

### 14.3 查看状态

```bash
zsh scripts/watch-tailscale-ssh.sh status
```

### 14.4 停止后台巡检

```bash
zsh scripts/watch-tailscale-ssh.sh stop
```

### 14.5 重新安装并启动

```bash
zsh scripts/watch-tailscale-ssh.sh restart
```

### 14.6 指定不同参数

```bash
zsh scripts/watch-tailscale-ssh.sh restart 100.82.42.75 22 180
```

参数含义：

- 第一个参数：目标 IP
- 第二个参数：目标端口
- 第三个参数：基础检查间隔，单位是秒

## 15. 为什么关闭终端后它仍然有效

因为它不是依赖某个终端窗口里的前台循环，而是注册成了 macOS 的 `launchd` 用户级 `LaunchAgent`。

这意味着：

- 关闭终端窗口不影响它
- 登录系统后它会自动启动
- 它会长期在后台运行

---

## 16. 为什么重启电脑后它还能继续工作

因为 `scripts/watch-tailscale-ssh.sh start` 会在：

```bash
~/Library/LaunchAgents/
```

下生成对应的 `plist` 文件，并交给 `launchd` 管理。

只要这个 `LaunchAgent` 没被停止或删除，系统下次登录后仍然会启动它。

同时，新的调度方式不是“脚本自己 `while true` + `sleep`”，而是：

- 由 `launchd` 按基础间隔唤起一次脚本
- 脚本读取状态文件，决定这次是否真的需要执行检查
- 状态文件按键值安全解析，而不是直接 `source`

这样做的一个重要好处是：

- 如果机器在睡眠期间错过了检查点，`launchd` 会在唤醒后尽快补跑一次

这比脚本自己在后台长期 `sleep` 更适合笔记本电脑。

---

## 17. 为什么这套方案对 CPU 影响很小

原因主要有三点：

- 基础检查频率低，默认是 3 分钟一次
- 每次检查执行的命令都很轻量
  - `tailscale ip -4`
  - `route`
  - `nc`
  - `tailscale ping`
  - 必要时才会调用 `tailscale down/up`
- 平时没有一个脚本在前台无限循环 `sleep`
- 当异常连续发生且确实触发重启时，检查间隔会自动翻倍，但有 1 小时上限
- 每天跨过 0 点后，检查间隔会恢复到基础值

因此，这不是一个高频忙轮询脚本，而是一个低负担、自适应的守护方案。

---

## 18. 这套方案的边界

这套方案适用于以下条件同时成立的场景：

- Tailscale 本身仍然能够运行
- 本机原生 `sshd` 已经开启
- 问题主要发生在 Shadowrocket 刷新或接管路由之后
- 重启 Tailscale 通常能恢复 SSH

它不适用于：

- Tailscale 账号本身掉线
- `tailscaled` 无法启动
- 本机 `sshd` 没开
- 远端主机的 `22` 端口根本没开，且你希望脚本替你修远端问题
- 更大范围的局域网、路由器或运营商故障
- 你需要对多个远端目标分别做不同的健康判断

所以它不是一个通用网络修复器，而是一个针对“同机多隧道路由争用”场景、且对远端故障保持保守判断的工程化修复方案。

---

## 19. 这份资料最重要的结论

这次问题最关键的结论有三条：

第一，问题不是“SSH 服务本身坏了”，而是：

- **系统里普通应用使用的路由，不再指向 Tailscale 的实际隧道接口**

第二，问题不是“Tailscale 与 Shadowrocket 协议冲突”，而是：

- **两者都在同一台 Mac 上争夺某些流量的路由归属权**

第三，实际排障时，最值得优先看的不是：

- `tailscale status`

而是：

- `route -n get <目标 Tailscale IP>`
- `nc -vz <目标 Tailscale IP> 22`

因为它们更直接回答：

- 目标流量到底走没走对路
- 你的原生 SSH 到底能不能用

只要这两项检查思路仍然有效，这套方案就仍然有可迁移价值。
